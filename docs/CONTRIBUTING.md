# Contributing & Maintainer Guide

For contributors making changes and maintainers cutting releases. See also: `AGENT-RULES.md` (plugin artifact rules) and `docs/rfcs/AGENT-RULES.md` (RFC rules).

---

## Quick start

### Prerequisites

| Tool | Purpose | macOS | Linux / WSL |
|---|---|---|---|
| `bats-core` | Hook and script tests | `brew install bats-core` | `sudo apt install bats` |
| `jq` | `scripts/package.sh` JSON parsing | `brew install jq` | `sudo apt install jq` |
| `python3` or `node` | Plugin manifest JSON validation | Usually pre-installed | Usually pre-installed |

### Run the test suite

```bash
# Unit tests (no @integration)
tests/run.sh

# Include integration tests
tests/run.sh --integration
```

Exit code = number of failed suites. CI runs `--integration`.

## Making changes

### Plugin artifacts (skills, hooks, commands, templates, agents)

Follow `AGENT-RULES.md` at the repo root. Key constraints:

- One artifact per commit. Don't bundle a skill change with a hook change.
- Skills need `name:` and `description:` YAML frontmatter; descriptions must list concrete trigger phrases.
- Hooks default to warn (exit 0). Block (exit 2) requires explicit justification.
- New hooks must be registered in `hooks/hooks.json`.
- New commands must appear in the README.md command table.

### RFCs

Follow `docs/rfcs/AGENT-RULES.md`. Short version: draft → second opinion → accepted → implement → archived. Don't implement before status is `accepted`.

### Counts and docs sync

When you add or remove a skill, command, hook, template, or agent, update the machine-readable counts block in `docs/references/_repo-context.md`:

```
<!-- validate-counts:start
skills=N
commands=N
hooks=N
templates=N
agents=N
validate-counts:end -->
```

`tests/plugin/validate_docs_sync.sh` checks these counts against disk on every `tests/run.sh` invocation. If the counts drift, the validator fails with `FAIL <type>: disk=X expected=Y`.

New commands also need a `/<name>` entry in the README.md command table — the validator checks for this too.

## Cutting a release

Full procedure: `docs/PACKAGING.md`. Short version:

1. Bump `version` in `.claude-plugin/plugin.json`
2. Preview: `scripts/package.sh --dry-run` — confirm no dev files in manifest
3. Commit the version bump to `main`
4. Tag: `git tag vX.Y.Z && git push origin vX.Y.Z`
5. Watch CI — `test` job runs first; `release` job only starts after it passes
6. Verify the GitHub Release has the attached `.tar.gz`
7. Verify `marketplace.json` on `main` was updated by the CI `[skip ci]` commit

Never push directly to the `release` branch — it is CI-owned and force-pushed on every release.

## What to check on every push to `main`

The `test` CI job runs `tests/run.sh --integration` automatically. These are the validators it runs:

| Validator | What it checks |
|---|---|
| `tests/plugin/validate_manifest.sh` | `plugin.json` is valid JSON + required fields + semver version |
| `tests/plugin/validate_docs_sync.sh` | Artifact counts on disk match `_repo-context.md`; all commands in README.md |
| `tests/skills/validate_frontmatter.sh` | Every `SKILL.md` has `name:` and `description:` frontmatter |
| `tests/templates/validate_structure.sh` | Every template has at least one markdown heading |
| `tests/hooks/*.bats` | Hook behavior: plan-gate, scope-check, adjacent-function, secret-scan, etc. |
| `tests/scripts/package.bats` | `package.sh` flag parsing, manifest exclusions, CI bug regressions |

If any validator fails locally, fix it before pushing — CI will catch the same failure.

### After adding an artifact

Checklist:

- [ ] Counts updated in `docs/references/_repo-context.md` validate-counts block
- [ ] New command listed in README.md command table (if applicable)
- [ ] Hook registered in `hooks/hooks.json` (if applicable)
- [ ] `tests/run.sh` passes locally

### After editing `docs/`

Checklist:

- [ ] `docs/README.md` file registry is current
- [ ] `docs/_index.json` `update_triggers` entries are current
- [ ] `_repo-context.md` "Last updated" date reflects the change
- [ ] Any RFC status change updates `docs/rfcs/AGENT-RULES.md` index entries

## Maintainer-only: release branch

The `release` branch is generated by `scripts/package.sh` and force-pushed by CI. It contains only distributable files — no `docs/`, `tests/`, `scripts/`, or `CLAUDE.md`. Consumers who install via `marketplace.json` receive the `release` branch content.

Do not manually commit to `release`. If something is wrong there, fix `main` and cut a new release.

For troubleshooting CI failures, see the Troubleshooting table in `docs/PACKAGING.md`.
